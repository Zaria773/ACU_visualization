# 抽签系统扩展功能策划 - "暗规则"与"成就系统"

根据用户反馈，我们将在基础的维度抽签之上，扩展出更有趣的元游戏（Meta-Game）玩法。核心理念是将抽签结果不仅仅作为一次性的随机事件，而是作为触发更复杂逻辑的“种子”。

## 1. 核心概念

### 1.1 "共鸣" (Resonance) - 有趣的话/成就显示
当满足特定条件时，抽签结果界面不再只显示干巴巴的档位，而是会弹出一个特殊的“共鸣”提示。
- **表现形式**：类似于游戏中的成就解锁弹窗，或者 RPG 里的特殊旁白。
- **内容**：一句有趣的话、一句吐槽、或者一段神秘的预言。

### 1.2 "暗流" (Undercurrent) - 隐藏提示词注入
当满足特定条件时，系统会**悄悄地**向 LLM 的上下文中注入一段隐藏提示词（Hidden Prompt）。
- **用户感知**：用户在界面上可能只看到一个神秘的图标闪烁，或者完全无感知。
- **效果**：改变剧情走向、触发特殊事件、改变 NPC 态度等。
- **类型**：
    - **正面 (Blessing)**：如“今天运气爆棚，路边捡到钱”。
    - **负面 (Curse)**：如“你总感觉背后有人盯着你”。
    - **恶搞 (Prank)**：如“所有 NPC 说话都带口癖”。

### 1.3 "命运之轮" (Wheel of Fate) - 统计与计数
系统会记录用户的抽签历史，作为触发上述规则的条件。
- **计数器**：记录每个维度、每个档位的抽取次数。
- **连击 (Streak)**：连续抽取同一档位的次数（如连续 3 次“大凶”）。
- **组合 (Combo)**：单次抽签中同时满足多个特定档位（如“大凶” + “理智崩坏”）。

---

## 2. 规则引擎设计 (Rule Engine)

我们需要一个灵活的规则引擎来定义触发条件。

### 2.1 触发条件 (Triggers)

支持逻辑运算符（AND, OR, NOT）的组合条件：

*   **当前结果 (Current Result)**:
    *   `dimension_id == 'luck' && tier_id == 'great_curse'` (运势是大凶)
    *   `dimension_id == 'sanity' && value < 10` (理智值极低)
*   **历史统计 (Statistics)**:
    *   `count('luck', 'great_curse') >= 10` (累计抽到 10 次大凶)
    *   `streak('luck', 'great_curse') >= 3` (连续 3 次大凶)
*   **复合条件 (Compound)**:
    *   `current('luck', 'great_blessing') && current('sanity', 'crazy')` (幸运的疯子)

### 2.2 执行动作 (Actions)

*   **ShowToast**: 显示“共鸣”文字（成就弹窗）。
*   **InjectPrompt**: 注入“暗流”提示词（可设定持续回合数）。
*   **PlaySound**: 播放特殊音效。
*   **UnlockFeature**: 解锁某个隐藏的 UI 功能（如更高级的抽签动画）。

## 3. 玩法示例 (Brainstorming)

### 3.1 简单的成就
*   **规则名**：非酋的凝视
*   **条件**：`streak('luck', 'great_curse') >= 3` (连续 3 次大凶)
*   **共鸣**：“你的脸黑得连黑洞都看不下去了...”
*   **暗流**：无

### 3.2 复合维度的化学反应
*   **规则名**：狂乱的幸运儿
*   **条件**：`current('luck', 'great_blessing')` AND `current('sanity', 'zero')` (大吉 + 理智归零)
*   **共鸣**：“疯子的运气总是特别好，不是吗？”
*   **暗流**：“[系统指令：接下来的剧情中，主角虽然行为疯癫，但所有鲁莽的举动都会意外地导致好的结果]”

### 3.3 累积效应
*   **规则名**：san值危机
*   **条件**：`count('sanity', 'low') >= 50` (累计 50 次低理智)
*   **共鸣**：“你已经凝视深渊太久了...”
*   **暗流**：(永久生效) “[系统指令：世界在主角眼中开始出现轻微的扭曲，偶尔会看到不存在的触手幻影]”

### 3.4 隐藏彩蛋
*   **规则名**：开发者的怜悯
*   **条件**：`streak('luck', 'great_curse') >= 10` (连续 10 次大凶)
*   **共鸣**：“太惨了，系统决定送你一个挂。”
*   **暗流**：“[系统指令：下一回合必定发生一个极小概率的奇迹]”

## 4. UI/UX 规划

### 4.1 规则编辑器 (Rule Editor)
需要一个新的子面板 `RuleManagerPanel`。
*   **可视化编辑器**：类似于 Scratch 或工作流编辑器的简化版。
*   **条件构建器**：下拉菜单选择维度、档位、比较符。
*   **动作配置**：输入显示的文字、注入的提示词。

### 4.2 统计面板 (Statistics Panel)
*   展示用户的“命运轨迹”。
*   饼图/柱状图显示各维度分布。
*   “成就墙”：点亮已触发的规则图标。

---

## 5. 开发路线调整

鉴于这个扩展功能的复杂性，建议分阶段实施：

**Phase 1: 基础架构 (Current Task)**
*   完成基础的维度管理 (`DimensionManagerPanel`)。
*   完成基础的抽签功能 (`DivinationPanel`)。
*   **预留接口**：在 `Dimension` 数据结构中预留 ID 字段，方便后续规则引用。

**Phase 2: 规则引擎 (Next Task)**
*   实现 `RuleStore`。
*   开发 `RuleEditor` 面板。
*   实现抽签后的规则匹配逻辑。

**Phase 3: 统计与成就**
*   实现数据持久化（统计数据）。
*   开发“成就墙” UI。
